# 统计页面设计文档

本文档描述 `quizit` App 中“学习统计”页面的设计，覆盖数据存储、统计计算逻辑、UI 控制与交互状态。

## 1. 页面目标

统计页的目标是让用户按“日期”查看学习行为汇总，并同时支持两类场景：

1. 回看历史：查看某一天的最终统计结果。
2. 查看当天：在页面内获取当日的实时统计，并在后续练习/测验数据补齐后自动回填。

页面入口来自顶部菜单的“学习统计”，路由为 `/stats`，只允许登录用户访问。

---

## 2. 数据存储设计

### 2.1 核心表：`daily_user_stats`

统计页的持久化数据保存在 `public.daily_user_stats` 表中。

字段含义如下：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `user_id` | `uuid` | 用户 ID |
| `date` | `date` | 统计日期，按北京时间自然日切分 |
| `questions_reviewed` | `int` | 当日测验题目数 |
| `question_time_spent` | `int` | 当日测验耗时，单位秒 |
| `quizzes` | `jsonb` | 按测验模板分组的题目数量 |
| `cards_reviewed` | `int` | 当日闪卡题目数 |
| `card_time_spent` | `int` | 当日闪卡耗时，单位秒 |
| `decks` | `jsonb` | 按 deck 分组的卡片数量 |
| `inserted_at` | `timestamptz` | 该条记录最后写入时间 |

主键为 `(user_id, date)`，因此同一个用户同一天只保留一条统计记录。

### 2.2 统计结果的粒度

`daily_user_stats` 是“按天聚合”的事实表，不存单次答题明细。单次明细仍然来自：

1. `card_reviews`：每次闪卡或测验作答都会写入。
2. `quiz_runs`：测验运行结果保留更高层级的回放数据。

统计页只消费聚合后的日统计，不直接从明细表做前端聚合。

### 2.3 分组字段的语义

`quizzes` 和 `decks` 都是 JSON 映射，key 为业务 ID，value 为该 ID 对应的计数。

典型形态：

```json
{
  "quiz-id-1": 12,
  "quiz-id-2": 5
}
```

```json
{
  "deck-id-1": 20,
  "deck-id-2": 3
}
```

页面显示时会二次查询 `quizzes` / `decks` 表，补全标题与删除状态。

### 2.4 时区口径

统计按北京时间 `Asia/Shanghai` 切日，不使用浏览器本地时区。

前端与 SQL 的约定是：

1. 所有统计日期使用 `YYYY-MM-DD` 字符串表示。
2. 计算“今天”时以北京时间日期为准。
3. 前端展示月历时，同样使用北京时间日期进行比较和筛选。

这能避免用户在跨时区环境下看到“日期错位”。

## 3. 统计计算逻辑

### 3.1 数据来源：`compute_daily_user_stats`

数据库函数 `public.compute_daily_user_stats(target_date, target_user)` 是统计页的统一计算入口。

它的作用是：

1. 从 `card_reviews` 中取出指定日期的所有记录。
2. 按 `user_id + 日期 + is_question + belongs_to` 分组。
3. 分别聚合出：
   - 闪卡统计：`cards_reviewed`、`card_time_spent`、`decks`
   - 测验统计：`questions_reviewed`、`question_time_spent`、`quizzes`
4. 再把两部分 full join 成一条日统计记录。

### 3.2 聚合规则

聚合口径如下：

1. `cards_reviewed` / `questions_reviewed`：
   - 对当日记录计数。
2. `card_time_spent` / `question_time_spent`：
   - 对 `time_spent` 求和，空值按 0 处理。
3. `decks` / `quizzes`：
   - 按 `belongs_to` 分组计数。
   - `belongs_to` 为空时不进入 JSON 聚合。

### 3.3 数据按需加载与回写

统计页做的是“数据按需加载”，不是前端代码分包懒加载：

1. `StatsPage` 在路由进入后才开始读取统计数据。
2. 只读取当前月份需要展示的日统计。
3. 只在选中“今天”时额外拉取实时统计。
4. 只在当前日期存在分组 ID 时，才补查 deck / quiz 的名称。

页面的数据流是“先读表，再按需回填”：

1. 首先从 `daily_user_stats` 读取当前月已有数据。
2. 对当前月内、今天及以前的缺失日期，调用 `backfillDailyStat(date)`。
3. `backfillDailyStat` 内部执行 `compute_daily_user_stats(target_date, target_user)`。
4. RPC 返回后，前端把结果 `upsert` 回 `daily_user_stats`。
5. 再将结果合并进本地 `monthStats` 缓存。

这个设计的目标是：

1. 月视图尽量只依赖聚合表，不直接扫明细表。
2. 历史月数据可复用，刷新页面后也不会丢失。
3. 当天数据仍然保留实时补齐能力。

### 3.4 “最终统计”判定

页面会根据 `inserted_at` 判断某天统计是否可视为最终结果。

判定规则如下：

1. 没有 `inserted_at`，视为非最终。
2. `inserted_at` 的北京时间日期晚于统计日期，视为最终。
3. `inserted_at` 的北京时间日期等于统计日期，不视为最终，因为当天仍可能继续变化。

这套规则主要用于：

1. 决定是否跳过回填。
2. 决定日历格是否显示“统计中”语义。
3. 避免把当天尚未稳定的数据当成最终值展示。

### 3.5 缺失日期回填

月视图加载完成后，前端会扫描当前月内、今天及以前的日期。

对每个候选日期：

1. 如果已有最终统计，跳过。
2. 如果已经回填过，跳过。
3. 如果正在回填，跳过。
4. 否则加入 `backfillDailyStat(date)` 的执行队列。

回填的目标不是补齐整段历史，而是补齐“当前可见月内尚未落表的日期”，让日历格保持可展示数据，同时避免重复扫描全量明细。

### 3.6 空数据占位

如果某一天计算结果为空，前端会生成一条零值记录：

1. `questions_reviewed = 0`
2. `question_time_spent = 0`
3. `cards_reviewed = 0`
4. `card_time_spent = 0`
5. `quizzes = null`
6. `decks = null`

这样做能保证：

1. 日历格子有统一的数据结构。
2. 查询月视图时不会因为缺行导致逻辑分支过多。

## 4. UI 控制设计

### 4.1 页面结构

统计页分为左右两栏：

1. 左侧：月历视图。
2. 右侧：选中日期的详情面板。

顶部提供月份切换与当前月份标题。

### 4.2 访问控制

页面初始化时先检查登录状态：

1. `authChecked = false` 时显示“正在加载用户信息”。
2. 没有 `userId` 时显示“请先登录以查看学习统计”。
3. 只有登录用户才能继续加载统计数据。

### 4.3 月历控制

月历按自然月渲染，显示周日到周六七列布局。

每个日期格子支持以下状态：

1. `今天`：特殊高亮。
2. `已选中`：绿色边框/背景强调。
3. `有数据`：显示闪卡数和测验题数。
4. `统计中`：显示“统计中”标签。
5. `无数据`：显示“无”并禁用点击。
6. `未来日期`：禁用点击。

月份切换规则：

1. 可以查看上一个月。
2. 不能切换到未来月份。
3. 切换月份后，默认选中该月第一天。
4. 如果切到当前月且第一天在未来，则回退到今天。

### 4.4 详情面板

右侧详情面板展示当前选中日期的两类统计：

1. 闪卡区块
2. 测验区块

每个区块都显示：

1. 总题数
2. 总耗时
3. 按 deck / quiz 的分组明细

如果分组明细存在，则按名称 + 数量列表展示；如果名称缺失，会先请求对应表补全。

### 4.5 当天刷新

当选中日期是今天时，右上角显示“刷新”按钮。

点击后会重新执行 `compute_daily_user_stats`，重新获取当天实时数据。

这个按钮主要用于：

1. 刚完成一轮练习后立即看到最新结果。
2. 减少页面刷新依赖。

### 4.6 名称补全与跳转

`decks` 与 `quizzes` 的分组 ID 会转换为可点击条目：

1. deck 条目跳转到 deck 编辑页。
2. quiz 条目跳转到 quiz run 详情页。

如果目标对象已删除，页面会保留名称展示，但不提供跳转。

## 5. 状态与数据流

### 5.1 组件状态

`StatsPage` 维护的核心状态包括：

1. `userId`：当前登录用户。
2. `month`：当前查看的月份。
3. `selectedDate`：当前选中的日期。
4. `monthStats`：当月统计缓存。
5. `todayLive`：当天实时统计。
6. `pendingBackfillDates`：正在回填中的日期集合。
7. `deckNames` / `quizNames`：分组名称缓存。

### 5.2 加载流程

页面进入后的加载顺序：

1. `supabase.auth.getUser()` 获取用户。
2. 读取当月 `daily_user_stats`。
3. 对当前月内、今天及以前的缺失日期执行回填。
4. 如果当前选中日期是今天，则拉取实时统计。
5. 右侧详情根据 `selectedDate` 渲染。

### 5.3 合并逻辑

页面渲染使用 `mergedStats = monthStats + todayLive` 的方式：

1. 月表中的数据先作为基础。
2. 如果当天存在实时数据，则覆盖当天对应条目。

因此：

1. 历史日期直接看持久化值。
2. 今天可以看到更及时的结果。
3. 月视图不会因为今天尚未落表而闪烁为空。

### 5.4 选择约束

日期点击有两个前置限制：

1. 不能选择未来日期。
2. 没有任何学习行为的日期不允许进入详情态。

这样能避免详情页出现无意义空状态，并让日历保持“只看有内容的日期”的交互语义。

## 6. 异常与兜底

### 6.1 读取失败

如果读取月数据失败：

1. 页面显示“加载统计失败”。
2. 控制台输出错误日志。

### 6.2 实时计算失败

如果当天实时统计 RPC 失败：

1. 页面显示“实时统计失败”。
2. 保留已加载的历史月数据，不影响其它日期。

### 6.3 名称解析失败

如果 `decks` 或 `quizzes` 名称查询失败：

1. 仍显示 ID 或 fallback 文本。
2. 不阻断主统计渲染。

## 7. 设计取舍

### 7.1 为什么采用“日聚合表 + 即时回填”

直接每次进页面都扫描 `card_reviews` 虽然实现简单，但会带来：

1. 月视图查询成本高。
2. 每次渲染都要重复聚合。
3. 历史日统计难以稳定复用。

采用 `daily_user_stats` 后：

1. 月视图可以快速读取。
2. 详情页切换更顺滑。
3. 当天数据仍可通过 RPC 保持实时性。

### 7.2 为什么前端负责 upsert

目前统计页在前端完成 RPC 结果写表，原因是：

1. 逻辑集中在页面侧，便于根据 UI 状态做增量回填。
2. 对当天数据的刷新与展示更直接。

代价是：

1. 页面存在一定数据写入责任。
2. 如果后续统计页流量变大，可能更适合迁移为服务端定时任务或边写边算的后端流程。

### 7.3 为什么强调“数据按需加载”

统计页的瓶颈不是页面代码本身，而是统计数据的读取范围。

如果一进入页面就拉全量历史，会带来：

1. 网络请求更重。
2. 回填逻辑更复杂。
3. 月切换时的状态更新更容易抖动。

因此这页的设计重点是：

1. 页面进入后只取当前月所需数据。
2. 今天单独处理实时层。
3. 缺失日只补当前可见范围。
4. 其余历史数据等用户切到对应月份时再加载。

## 8. 可扩展方向

后续如果需要增强统计页，可以考虑：

1. 增加周视图或趋势图。
2. 增加“连续学习天数”与“总学习时长”。
3. 把 `quizzes` / `decks` 进一步扩展为包含标题快照，减少名称回查。
4. 把回填逻辑下沉到后台任务，降低前端写表责任。
5. 对月视图做服务端分页或缓存，支撑更大数据量。
